
## 테스트 코드는 왜 필요할까
---

> 내가 생각하는 유지보수하기 쉬운 소프트웨어 == '파악하기 쉬운' 소프트웨어
> 파악 ?
> 1. 어떤 코드를 수정했을때의 의도치 않은 영향도를 파악 (라이브러리/프레임워크 버전 변경도 포함)
> 2. 예상치 못한 문제 발생시 원인 파악
> 3. 이 소프트웨어가 어떤 기능들을 수행하는지, 어떤 부분이 중요한지에 대한 파악

- 테스트 코드는 일종의:
  - '안전망'으로서:
    - 코드 수정 시 의도치 않은 영향도를 파악하게 해주고 (1번)
    - 예상 가능한 문제들을 사전에 방지함으로써, 예상치 못한 문제 발생 시 원인 범위를 좁혀준다 (2번)
  - '기능 명세서'로서:
    - 소프트웨어가 어떤 기능을 수행하는지 이해하게 해주며 (3번)
라고 생각한다.

- 특히나 앞으로는 AI 에이전트에게 개발(특히 코드를 작성하는 부분)을 더욱 위임할 것으로 예상되는 상황에서, 테스트 코드를 통해
  - 인간은 : AI가 작성한 코드에 대한 불안감을 덜고, AI와 함께 견고한 소프트웨어를 만들어나갈 수 있고,
  - AI는 : 좀 더 효율적으로 소프트웨어를 파악하고 디버깅할 수 있지 않을까 하는 생각이 든다.

- 따라서, 앞으로 AI 에이전트에게 코드 작성을 점점 더 위임하게 된다면, 사람은 구현 자체보다는 **요구사항을 명확히 정의하고 테스트로 이를 검증하는 구조를 만드는 일**에 더 많은 신경을 쓰게 되지 않을까 생각한다.

> 즉, 앞으로 테스트 코드는 인간과 AI 모두에게 중요한 이해의 인터페이스가 될 것 같다.

## 일반적인 웹 애플리케이션에서 계층별 테스트 코드 생각해보기
---

### Controller
> Client(호출자)와 맞닿은 인터페이스

**안전망**
- 직렬화 / 역직렬화 검증
- 인증 / 인가 검증
- 시스템에 큰 부하를 주는 입력값은 허용하지 않는다.
  - 너무 긴 기간을 조회
  - 페이징 사이즈가 너무 큼
  - 등등

**기능 명세서**
- Request / Response
  - 포맷
  - 요청(정상/비정상 등)에 따라 어떻게 응답할지
  - 정상적이지 않은 시스템 환경일때 어떻게 응답할지
  - 등등

```
그리고 "정상적이지 않은 시스템 환경일 때 어떻게 응답할지"라는 부분은 Controller 테스트만으로 커버하기보다 Circuit Breaker나 Fallback 로직이 있다면 해당 레이어에서 검증하는 게 더 명확할 때도 있어요.
```

### Service
> 기능 단위의 요구사항(?) 구현을 위해, 도메인 로직, db 호출, 외부 api 호출 등의 흐름을 조합

**안전망**
- (요구사항이 바뀌지 않는한) 구현의 변경으로 인해, 요구사항이 틀어지지 않게
- 즉, 이 기능 단위의 요구사항에서 **반드시 보장돼야하는 부분을 판단**하는게 중요하다고 생각.
  - 정상적인 상황에서 반드시 보장돼야하는 부분
  - 비정상적인 상황에서 반드시 보장돼야하는 부분

  -> 반드시 보장돼야하는 부분들 ?
  -> 음 ,, '상품권 사용 처리'를 생각해보면 '마일리지 적립 api 호출'은 반드시 필요함.
  -> 10번 넘게 시도했을때 제대로 블로킹되는지 이런 부분들은 반드시 정도까지는 아니라고 생각, 경우에 따라서는 운영에서 이런게 매우 중요한 상황이라고 한다면, 검증해야 한다고 생각. **즉 맥락에 따라 달라질 수 있다고 생각.**
  -> '운영자에게 알림' 등도 마찬가지 -> 어떤 이벤트가 발생했을 때, 적절한 대응을 하는 자동화된 시스템이 있고, 운영자에게 이를 알리는건 참고용 정도인 느낌인 경우엔 '운영자에게 알림'까지 테스트를 작성할 필요가 없을 수 있지만, 아직 초기이고 이슈 발생시 운영에서 매우 중요한 대응들을 한다면 이 부분에 대한 테스트는 필수일 것으로 생각.

- 정산 건별내역
  - 정상적인 상황
    - 해당 결제 수단에 대한 계약 조건이 관리돼야한다.
    => 계약 조건에 따른 수수료 계산, 지급일자 계산이 정확하게 돼야한다.
  - 비정상적인 상황
    - 계약조건이 관리되지 않는다
    - 해당 결제수단이 관리되지 않는다
    => 운영자에게 반드시 '알림'이 가야한다.


**기능 명세서**
- Service는 Controller처럼 input/output이 명확하진 않는데, 어떤 관점에서 생각해보면 좋을까 ?
- 이 기능 단위에서 구현하고자 하는 '요구사항'이 무엇인지 ? => 이걸 어떻게 나타내지 ?
  - 결국 안전망을 구축하는게 어떤 요구사항을 만족해야하는 것인지를 표현하는 것이기 때문에, 이 부분까지 포함되는 것이라고 생각

### 도메인 모델
> Service 레이어에서 나열하는 것과, 도메인 모델 내로 캡슐화하는 것 ?
> 어떤 부분을 캡슐화하는게 좋을까 ?
> Service 레이어는 어떤 흐름을 조합하는 관점이라면, 도메인 모델은 도메인 관련 로직(수수료 계산, 상태 변경 등)을 응집해놓는 관점 ?
> ex : 요구사항 => 결제 내역을 정산 가능한 건별 내역으로 만들기
> - Service 레이어 (요구사항 구현 흐름) : 기존에 만들어진거 삭제 -> 해당 결제가 발생한 가맹점 및 결제 수단 확인 -> 매핑된 계약 조건 가져오기 -> 정산을 위한 계산
> - 도메인 모델 : 결제 수단, 계약 조건, 정산 내역 => 결제 수단, 계약 조건을 갖고 수수료 계산 등

**안전망**
- 도메인 로직이 의도한대로 동작한다
  - 도메인 로직 : 요구사항 자체일 수도 있고, 요구사항 구현의 한 부분일 수도 있고
- 도메인 모델 테스트의 진짜 가치는 "이 객체는 항상 유효한 상태를 유지한다"는 걸 보장하는 것
  - ex : 정산 내역은 수수료율이 음수가 될 수 없다든지, 계약 조건 없이는 정산 내역이 생성될 수 없다든지. 이런 불변식이 도메인 모델 테스트에서 명세화되면, Service 레이어는 흐름에만 집중할 수 있음


**기능 명세서**
- 도메인 로직을 검증하는 것 자체가, 도메인 모델별로 어떤 부분을 담당하고 있는 것인지에 대한 명세가 된다고 생각

### Repository
> DataSource와 맞닿은 인터페이스
> JPA처럼 EntityManager, 영속성 컨텍스트가 많은 부분을 담당하는 경우는 (즉, 내 제어권을 벗어난 경우는) ??

**안전망**
- Read
  - 의도한 데이터만 조회하는지
  - 데이터가 많아질수록 성능 저하는 없는지

- Write
  - 의도한 데이터만 변경하는지

```
제어권 밖(EntityManager)의 검증: JPA가 쿼리를 잘 짜주는지 믿는 것도 좋지만, **'내가 생각한 타이밍에 쿼리가 나가는가?'**와 **'N+1 문제가 발생하지 않는가?'**는 Repository(또는 통합 테스트)에서 검증해야 할 안전망입니다.

복잡한 쿼리: Querydsl 등으로 복잡한 동적 쿼리를 작성한다면, 이는 명백히 Repository 테스트에서 데이터 정합성을 확인해야 합니다.


성능 저하 검증을 언급한 게 좋아요. 다만 현실적으로는 테스트 환경의 데이터 규모가 운영과 다르기 때문에, 이건 테스트보다 인덱스 설계 리뷰나 쿼리 실행계획 확인으로 보완하는 게 더 실용적인 경우가 많아요.
추가로, JPA 영속성 컨텍스트 관련해서 언급하셨는데 — N+1 문제처럼 의도치 않은 추가 쿼리 발생 여부를 검증하는 것도 Repository 테스트의 중요한 역할이에요. SQL 카운트 검증 같은 방식으로 접근할 수 있고요.
```
